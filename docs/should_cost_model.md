# Should-Cost / Clean-Sheet Cost Model — specification

**Status:** spec (Phase 0). No code yet — this is the IP on paper, reviewed before the
`costing` domain is built.
**Question it answers:** *"A vendor quoted €X for this server. What should it actually
cost — component by component, indexed to commodity markets — so we negotiate from our
number, not theirs?"*

This is the deterministic core. An LLM may later *propose* a bill of materials from a quote
PDF, but the cost numbers a buyer takes into a negotiation are always computed by tested
code — never generated by a model.

---

## 1. The discipline: 5-element clean-sheet teardown

Anchored to the standard clean-sheet cost build-up so it is defensible, not invented (the
same "no ad-hoc KPIs" rule that anchors the Power-BI metrics to SCOR/WMAPE):

1. **Direct material** — driven by a commodity index.
2. **Direct labour / conversion** — assembly + test.
3. **Manufacturing overhead** (burden) — applied as a % of material.
4. **SG&A** — selling, general & admin, as a % of works cost.
5. **Profit / margin** — the supplier's expected markup.

Elements 1–3 build a **component cost**; rolling those up and adding 4–5 gives the
**should-cost floor** and a **target price**.

---

## 2. Two costing methods (the honesty that makes it survive a vendor)

Not every component tracks a public commodity. Each `ComponentClass` declares its method:

| Method | Used for | How the "should-cost" is derived |
| --- | --- | --- |
| `teardown` | DRAM, NAND, copper chassis, aluminium, PCB, PSU | True index-driven material teardown (§3). Material moves with the commodity index. |
| `reference_price` | CPU, GPU | Silicon does **not** track a public commodity cleanly. The benchmark is **vendor list price × expected discount _band_**, not a fabricated material teardown. |

Labelling CPU/GPU as a reference-price *benchmark* (not a teardown) is deliberate: a fake
silicon teardown is the first thing a supplier's cost engineer would tear apart. The
benchmark-band framing is more credible and still produces a negotiation anchor.

**Discount bands are ranges, tied to a stated volume/supply tier** — a point estimate looks
invented; a band with an assumption looks like procurement. Assumption tag:
*"at our volume tier, normalized supply."*

| Class | Discount band | Notes |
| --- | --- | --- |
| CPU | **15–30%** | More leverage than GPU; multi-source-able, list discounting is routine at volume. |
| GPU | **0–15%** | Least leverage. In the supply-constrained market this model is set in (chip/memory crunch), GPU discount collapses toward **0** — sometimes list-or-above just for allocation. Default to the **bottom of the band** unless allocation is comfortable. |

This is itself the insight: **silicon is where you have the least leverage**, which is
exactly why the commodity-teardown components (memory, storage, metal, PCB) are where the
real savings live.

---

## 3. Per-component build-up (`teardown` method)

```
material_now    = base_material_cost × (index_now / index_baseline)
component_cost  = material_now
                + conversion_cost
                + material_now × overhead_pct
component_floor = component_cost × qty
```

- `base_material_cost` — the component's material cost at the baseline index date.
- `index_now / index_baseline` — the commodity multiplier as-of the run date. At baseline
  this is 1.0 (floor == base).
- `conversion_cost` — direct labour + test per unit (does not scale with the commodity).
- `overhead_pct` — manufacturing burden, applied to material only.
- `qty` — units of this component in the config.

### `reference_price` method (CPU/GPU)

```
component_floor = list_price × (1 − expected_discount_pct) × qty
```

No commodity multiplier, no overhead build-up — it is a negotiated-price benchmark, and the
breakdown labels it as such so it is never confused with a material teardown.

---

## 4. Config roll-up

```
material_total       = Σ component_floor                          # all lines (both methods)
assembly_integration = material_total × integration_pct           # system assembly + integration
sga                  = (material_total + assembly_integration) × sga_pct
should_cost_floor    = material_total + assembly_integration + sga
target_price         = should_cost_floor × (1 + target_margin_pct)
```

- `integration_pct`, `sga_pct`, `target_margin_pct` are the tunable knobs (`CostParams`),
  per-BOM or a global default.
- **Floor vs target:** `should_cost_floor` is the supplier's cost; `target_price` is the
  fair price we should be willing to pay (floor + a reasonable margin we concede). The
  negotiation gap is measured against `target_price`, not the bare floor.

---

## 5. Negotiation gap (the output that matters)

Two gaps are computed and **both are kept** — the headline is measured against
`target_price`, with the bare floor as a secondary diagnostic.

```
# Headline — realistic addressable savings (vs a fair price the supplier would accept)
gap_to_target_abs  = quoted_price − target_price
gap_to_target_pct  = gap_to_target_abs / quoted_price
addressable_saving = gap_to_target_abs × annual_volume

# Backstop — theoretical max / total margin stacked in the quote (vs cost-with-zero-margin)
gap_to_floor_abs   = quoted_price − should_cost_floor
gap_to_floor_pct   = gap_to_floor_abs / quoted_price
```

- `quoted_price` — the preferred `ProductSupplier` contract price (reuses the existing
  spend-provenance chain; the gap is queryable against real spend).
- **Why headline = vs target, not vs floor.** The floor is cost-with-zero-margin, and no
  supplier sells at cost. Walking in with *"your cost is €16.5k, so I'll pay €16.5k"* loses
  the room in the first sentence. `target_price` (floor + a fair conceded margin) is the
  number that survives: *"here's a price that gives you a reasonable margin — and you're
  still €X above even that."* Showing you expect the supplier to make margin is what makes
  the ask credible.
- **Why keep the floor anyway.** `gap_to_floor` is the total margin stacked in the quote —
  the theoretical ceiling. It is not a number you demand; it is a **ranking signal**: the
  categories with the fattest gap-to-floor are where the margin lives, so it tells you where
  to spend negotiation energy. On the chart the floor is a **backstop reference line behind
  the target**, never the headline figure.
- `addressable_saving` (vs target) scales the per-unit headline gap by annual volume — the
  €-figure that goes on the slide.

## 6. Commodity sensitivity (the killer view)

Recompute `should_cost_floor` at `index × (1 ± delta)` for the commodity-tracked lines, to
show how the floor moves with DRAM/NAND prices:

```
floor_low  = should_cost_floor evaluated at index_now × (1 − delta)
floor_high = should_cost_floor evaluated at index_now × (1 + delta)
```

This ties to the story the Power-BI repo already tells — *memory prices rose ~4× Sep–Nov
2025*. A static reorder point fails on that; a static should-cost does too. This one moves
with the market, so the negotiation floor is always current.

## 7. Edge cases the engine must handle (parity rigour with `lifecycle.py`)

| Case | Required behaviour |
| --- | --- |
| `qty = 0` | Component contributes 0; no divide-by-zero; line still appears in breakdown. |
| Missing commodity for a `teardown` line | Reject (422) — a teardown line must have a driver. Do not silently treat as reference-price. |
| `as_of` date before the commodity baseline | Reject (422) — cannot index before the series starts. |
| No price point exactly on `as_of` | Use the most recent point on or before `as_of` (step function, not interpolation) — documented and deterministic. |
| `reference_price` line with a commodity attached | Ignore the commodity (method wins); the breakdown notes it. |
| `quoted_price` missing | `should-cost` still computes; `cost-gap` returns null gap with a clear "no quote on file" flag. |
| All-`reference_price` BOM | Valid — `material_total` is just the sum of benchmarks; sensitivity returns flat (no commodity exposure). |

## 8. Worked examples

**Lead with the config where the defensible part carries the weight.** The hero example is
a memory/storage-dense node: its floor is built mostly from commodity-tracked components
with citable anchors (DRAM/NAND/steel/copper), so the "defensible cost floor" really is
defensible — and the commodity-sensitivity slide actually moves with the market. The
GPU-heavy box follows as a deliberately honest *secondary* example: the one place we have
the least leverage.

### 8a. Hero — storage/memory-dense node (commodity-driven floor)

A 4U storage appliance: heavy RAM + NVMe, a single modest CPU. Run indices reflect the
crunch — **DRAM ×1.8, NAND ×1.5**; metal/PCB at baseline.

| Line | Method | Detail | Calc | `component_floor` |
| --- | --- | --- | --- | --- |
| 64GB DDR5 RDIMM ×24 | teardown | base €110, conv €6, ovh 12%, **DRAM ×1.8** | (110×1.8 + 6 + 110×1.8×0.12) ×24 | **€5,466.24** |
| NVMe TLC module ×24 | teardown | base €90, conv €4, ovh 10%, **NAND ×1.5** | (90×1.5 + 4 + 90×1.5×0.10) ×24 | **€3,660.00** |
| 4U chassis ×1 | teardown | base €180, conv €40, ovh 10%, steel ×1.0 | 180 + 40 + 18 | **€238.00** |
| 2400W PSU ×2 | teardown | base €120, conv €18, ovh 10%, copper ×1.05 | (126 + 18 + 12.6) ×2 | **€313.20** |
| Motherboard ×1 | teardown | base €420, conv €60, ovh 10%, PCB ×1.0 | 420 + 60 + 42 | **€522.00** |
| ConnectX-7 NIC ×1 | teardown | base €700, conv €40, ovh 10%, PCB ×1.0 | 700 + 40 + 70 | **€810.00** |
| EPYC 9554 CPU ×1 | reference_price | list €7,120, discount 22% (mid CPU band) | 7120×0.78 | **€5,553.60** |

> **66% of material is commodity-tracked** (DRAM 33% + NAND 22% + metal/PCB ~11%); the single
> reference-price silicon line is only 34%. So the floor rests on citable anchors, not the OEM
> haircut — and the sensitivity view below has something to move.

```
material_total       = 5,466.24 + 3,660.00 + 238.00 + 313.20 + 522.00 + 810.00 + 5,553.60 = 16,563.04
assembly_integration = 16,563.04 × 0.06  = 993.78
sga                  = (16,563.04 + 993.78) × 0.08 = 1,404.55
should_cost_floor    = 16,563.04 + 993.78 + 1,404.55 = 18,961.37
target_price         = 18,961.37 × (1 + 0.10) = 20,857.51

quoted_price (preferred ProductSupplier) = 24,650.00

# Headline — vs target
gap_to_target_abs  = 24,650.00 − 20,857.51 = 3,792.49
gap_to_target_pct  = 3,792.49 / 24,650.00 = 15.4%
addressable_saving = 3,792.49 × 250 units/yr = €948,123

# Backstop — vs floor (total margin stacked; a ranking signal, not a demand)
gap_to_floor_abs   = 24,650.00 − 18,961.37 = 5,688.63
gap_to_floor_pct   = 5,688.63 / 24,650.00 = 23.1%
```

**Commodity sensitivity (the killer slide — and here it bites).** Move DRAM **and** NAND
±20% together:

```
should_cost_floor @ −20% = 16,927      # DRAM ×1.44, NAND ×1.20
should_cost_floor @ base = 18,961
should_cost_floor @ +20% = 20,996      # DRAM ×2.16, NAND ×1.80
swing ≈ ±€2,035  (±10.7% of the floor)
```

This is the narrative the Power-BI repo already tells — *memory ~4× Sep–Nov 2025*. On this
box the floor tracks the market: a static should-cost would be wrong by thousands per unit
as DRAM/NAND move; ours re-prices with the index. **This is the config to demo first.**

### 8b. Secondary — GPU/CPU-heavy box (where we have the least leverage)

Same engine, opposite shape: a compute node dominated by silicon.

| Line | Method | Detail | `component_floor` |
| --- | --- | --- | --- |
| 64GB DDR5 RDIMM ×8 | teardown | DRAM ×1.8 | €1,822.08 |
| 2U chassis ×1 | teardown | steel ×1.0 | €184.00 |
| 2400W PSU ×2 | teardown | copper ×1.05 | €313.20 |
| ConnectX-7 NIC ×1 | teardown | PCB ×1.0 | €810.00 |
| EPYC 9554 CPU ×2 | reference_price | 22% (mid CPU band) | €11,107.20 |
| L40S GPU ×1 | reference_price | **0%** (band-bottom — allocation market) | €6,800.00 |

```
should_cost_floor = 24,082.56   target = 26,490.82   quote = 30,500.00
gap_to_target = €4,009.18 (13.1%)     gap_to_floor = €6,417.44 (21.0%)
```

**Framed honestly:** ~85% of this box's material is the two reference-price silicon lines —
a *benchmark*, not a teardown — and the GPU sits at the bottom of its 0–15% band because in
an allocation market you pay near-list for silicon. So on this config the commodity-
sensitivity swing is negligible (~€340), and that is *the point*: silicon is where leverage
is weakest, which is exactly why the commodity-teardown components are where real,
defensible savings live. This box is a credible *second* slide — it shows we know the limits
of our own model. It is the wrong *first* slide, because it would demo the floor with its
most defensible feature (commodity tracking) switched off.

## 9. Commodity catalog + component-class methods (the seed decisions)

Phase-4 seed will lay down these commodities (synthetic, labelled), each with a 12-month
price series and one deliberate spike (memory ~4×, to give §6 a story):

| Commodity | Unit | Baseline | Drives |
| --- | --- | --- | --- |
| `DRAM_DDR5` | €/GB | 1.0 (normalised) | MEMORY |
| `NAND_TLC` | €/GB | 1.0 | STORAGE |
| `STEEL_CR` | index | 1.0 | CHASSIS (cold-rolled sheet metal — the bulk of a chassis) |
| `ALU_LME` | index | 1.0 | heatsinks / premium chassis |
| `COPPER_LME` | index | 1.0 | PSU, copper in chassis/cabling |
| `PCB_LABOR` | index | 1.0 | MOTHERBOARD, NIC |

> **Steel, not aluminium, for the chassis.** Server chassis are predominantly **cold-rolled
> steel**; aluminium is heatsinks and premium chassis. Pointing CHASSIS at copper/aluminium
> mislabels where the metal cost actually sits — so the chassis driver is `STEEL_CR`. This is
> the right resolution to stop at: gold connectors, tantalum caps, etc. have no clean public
> index and no story payoff — DRAM / NAND / steel / aluminium / copper / PCB is the floor of
> useful granularity.

| ComponentClass | Method | Commodity driver |
| --- | --- | --- |
| MEMORY | teardown | DRAM_DDR5 |
| STORAGE | teardown | NAND_TLC |
| CHASSIS | teardown | STEEL_CR |
| PSU | teardown | COPPER_LME |
| MOTHERBOARD | teardown | PCB_LABOR |
| NIC | teardown | PCB_LABOR |
| CPU | reference_price | — |
| GPU | reference_price | — |

Baselines are normalised to 1.0 so the multiplier is transparent; the spike lives in the
price *series*, not the baseline.

## 10. Defensibility summary (so it survives scrutiny)

- **Synthetic, labelled.** Commodity series are synthetic for the demo — stated plainly,
  same as the rest of the Power-BI repo. The architecture supports a real feed
  (TrendForce/DRAMeXchange for memory, LME for metals) later; those are paid, so they are a
  pilot-phase line item, not a capstone blocker.
- **Reference-price honesty for CPU/GPU.** No fake silicon teardown — a benchmark band.
- **Deterministic core.** The negotiation number is computed by tested code (`services/
  costing.py`, parity with `lifecycle.py`), not produced by an LLM.
- **Anchored, not ad-hoc.** Five-element clean-sheet build-up is a recognised costing
  discipline; the Power-BI page groups its measures under a "Clean-Sheet / Should-Cost"
  framework heading, same as SCOR for the rest.
